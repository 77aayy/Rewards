# قائمة المراجعة بعد التنفيذ
## Mandatory Review Checklist (لا تُعتبر المهمة منتهية بدونها)

**القاعدة:** بعد أي تنفيذ (feature أو إصلاح)، نفّذ هذه المراجعة قبل اعتبار المهمة "تم".

**الهدف:** المراجعة والفحص يغطيان **الحالات الحدّية** والتفاصيل التي تسبب **تعارضاً أو سلوكاً خاطئاً** — مراجعة شاملة **لكشف العيوب وتقليل الأخطاء**. التنفيذ الأول غالباً يغطي المسار السعيد فقط؛ المراجعة تكتشف أخطاء التهريب والسلوك الناقص وعدم تطابق المعرّفات.

---

### التاسك (Task)
- [ ] **مراجعة شاملة:** بعد كتابة الكود، تنفيذ كل البنود أدناه قبل القول "تم".
- [ ] **كشف العيوب:** التركيز على الحالات الحدّية (فارغ، تحميل، قيم غير صالحة) والتفاصيل التي تسبب تعارض (IDs، تهريب، طباعة قبل التحميل).
- [ ] **تقليل الأخطاء:** إصلاح أي عيب يُكتشف أثناء المراجعة — لا تأكيد بدون تحقق فعلي.

---

### 1. التهريب والرموز الخاصة (Escaping & special characters)
- [ ] المعرّفات (IDs) وخصائص `onclick` والقيم من المستخدم: التأكد أن الاقتباس (`'` و `"`) والرموز الخاصة لا تكسر الـ HTML/JS.
- [ ] مثال: `periodId` في الـ onclick وفي `getElementById` — استخدم تهريب آمن (مثلاً `periodIdSafe` للـ id و `periodIdEsc` للـ attribute).
- [ ] أسماء الفروع أو النصوص المعروضة: تهريب للـ HTML عند الإدراج في الـ DOM (مثلاً `<` → `&lt;`، `"` → `&quot;`) لتجنب XSS أو كسر الـ attribute.

---

### 2. الحالات الحدّية (Edge cases) — تغطية التفاصيل التي تسبب تعارض أو سلوك خاطئ
- [ ] حالة فارغة (لا بيانات): هل الواجهة لا تنكسر؟ رسالة مناسبة؟
- [ ] حالة التحميل: هل "جاري التحميل" أو الشريط يظهر؟ هل الطباعة لا تُفتح قبل اكتمال التحميل؟ (منع طباعة "جاري التحميل" بدل المحتوى.)
- [ ] إدخال غير صالح أو قيم null/undefined: هل يوجد تحقق أو قيمة افتراضية؟
- [ ] بيانات غير متوقعة (مثلاً اسم فرع فيه `"` أو `<`): هل التهريب يمنع كسر الـ DOM أو الـ attribute؟
- [ ] سلوك عند الإخفاء/الإظهار (مثلاً بطاقة مطوية): هل المحتوى يُحمّل حتى لو مخفي؟ هل الطباعة تأخذ المحتوى الصحيح؟

---

### 3. عدم نسيان سلوك (No forgotten behaviour)
- [ ] إذا الطلب كان فيه أكثر من سلوك (مثلاً "فتح البطاقة" + "عرض في الأعلى") — التأكد أن لا شيء حُذف عند التعديل.
- [ ] استرجاع أي زر أو رابط كان موجوداً قبل التعديل إن لزم.

---

### 4. التوافق (Consistency)
- [ ] المعرّفات والمحدّدات في الـ HTML = نفسها في الـ JS (مثلاً `periodId` vs `periodIdSafe` — يجب تطابق الاستخدام).
- [ ] أي dynamic ID يُبنى بنفس القاعدة في البناء وفي الاستدعاء.

---

### 5. الطباعة / التصدير (Print / export)
- [ ] إذا الميزة تتضمن طباعة أو تصدير: التأكد أن المحتوى المطبوع = المطلوب فقط.
- [ ] إخفاء أزرار الإجراء (مثل ⓘ وزر الطباعة) من التقرير المطبوع (مثلاً `no-print` أو CSS للطباعة).

---

### الخطوة العملية (Process) — مراجعة شاملة لكشف العيوب
بعد كتابة الكود:
1. **المسار الكامل:** من الضغط/الإدخال حتى النتيجة — هل كل خطوة تعمل؟
2. **الحالات الحدّية:** فارغ، تحميل، طباعة قبل التحميل، قيم غير صالحة — هل يوجد حماية أو رسالة مناسبة؟
3. **الـ DOM والمعرّفات والتهريب:** هل الـ IDs متطابقة بين HTML و JS؟ هل القيم المعروضة أو المُمرَّرة مُهَرَّبة؟
4. **الطباعة/التصدير:** إن وُجدت — هل المحتوى المطبوع صحيح؟ هل أزرار الإجراء مخفية؟
5. **عدم نسيان السلوك:** هل كل سلوك مطلوب (فتح، طباعة، عرض في الأعلى...) ما زال موجوداً؟
6. **إصلاح قبل "تم":** لا تأكيد بدون تحقق — إصلاح أي عيب يُكتشف ثم إعادة المراجعة إن لزم.

---

*هذا الملف ثابت في المشروع — لا يُحذف عند إغلاق البرنامج. راجعه بعد كل تنفيذ. التاسك موجود أيضاً في `TASKS.md` تحت "مراجعة إلزامية بعد التنفيذ".*
